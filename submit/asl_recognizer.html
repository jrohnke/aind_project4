Introduction¶

The overall goal of this project is to build a word recognizer for American Sign Language video sequences, demonstrating the power of probabalistic models. In particular, this project employs hidden Markov models (HMM's) to analyze a series of measurements taken from videos of American Sign Language (ASL) collected for research (see the RWTH-BOSTON-104 Database). In this video, the right-hand x and y locations are plotted as the speaker signs the sentence.

The raw data, train, and test sets are pre-defined. You will derive a variety of feature sets (explored in Part 1), as well as implement three different model selection criterion to determine the optimal number of hidden states for each word model (explored in Part 2). Finally, in Part 3 you will implement the recognizer and compare the effects the different combinations of feature sets and model selection criteria.

At the end of each Part, complete the submission cells with implementations, answer all questions, and pass the unit tests. Then submit the completed notebook for review!

  • normalized Cartesian coordinates

    • use mean and standard deviation statistics and the standard score equation to account for speakers with different heights and arm length

  • polar coordinates

    • calculate polar coordinates with Cartesian to polar equations
    • use the np.arctan2 function and swap the x and y axes to move the $0$ to $2\pi$ discontinuity to 12 o'clock instead of 3 o'clock; in other words, the normal break in radians value from $0$ to $2\pi$ occurs directly to the left of the speaker's nose, which may be in the signing area and interfere with results. Question 1: What custom features did you choose for the features_custom set and why?

Answer 1:

I started by following the suggestion to normalise the polar coordinate, this way systematic differences between speakers will be removed from the data, making signs from different speakers more comparable.

I added the distance between the left and right hand, split into x, y and total distance. This could help to differentiate between signs where the hands are close together or far apart. I also added a feature to see if the hands are moving in the same direction or mirror each other rather than moving independently. This should, for example, help identifying signs like the two handed cat, where both hands perform an identical but mirrored movement.

import math
from matplotlib import (cm, pyplot as plt, mlab)

def visualize(word, model):
    """ visualize the input model for a particular word """
    variance=np.array([np.diag(model.covars_[i]) for i in range(model.n_components)])
    figures = []
    for parm_idx in range(len(model.means_[0])):
        xmin = int(min(model.means_[:,parm_idx]) - max(variance[:,parm_idx]))
        xmax = int(max(model.means_[:,parm_idx]) + max(variance[:,parm_idx]))
        fig, axs = plt.subplots(model.n_components, sharex=True, sharey=False)
        colours = cm.rainbow(np.linspace(0, 1, model.n_components))
        for i, (ax, colour) in enumerate(zip(axs, colours)):
            x = np.linspace(xmin, xmax, 100)
            mu = model.means_[i,parm_idx]
            sigma = math.sqrt(np.diag(model.covars_[i])[parm_idx])
            ax.plot(x, mlab.normpdf(x, mu, sigma), c=colour)
            ax.set_title("{} feature {} hidden state #{}".format(word, parm_idx, i))

            ax.grid(True)
        figures.append(plt)
    for p in figures:
        p.show()

Cross-validation folds¶

If we simply score the model with the Log Likelihood calculated from the feature sequences it has been trained on, we should expect that more complex models will have higher likelihoods. However, that doesn't tell us which would have a better likelihood score on unseen data. The model will likely be overfit as complexity is added. To estimate which topology model is better using only the training data, we can compare scores using cross-validation. One technique for cross-validation is to break the training set into "folds" and rotate which fold is left out of training. The "left out" fold scored. This gives us a proxy method of finding the best model to use on "unseen data". In the following example, a set of word sequences is broken into three folds using the scikit-learn Kfold class object. When you implement SelectorCV, you will use this technique.

Scoring models with other criterion¶

Scoring model topologies with BIC balances fit and complexity within the training set for each word. In the BIC equation, a penalty term penalizes complexity to avoid overfitting, so that it is not necessary to also use cross-validation in the selection process. There are a number of references on the internet for this criterion. These slides include a formula you may find helpful for your implementation.

The advantages of scoring model topologies with DIC over BIC are presented by Alain Biem in this reference (also found here). DIC scores the discriminant ability of a training set for one word against competing words. Instead of a penalty term for complexity, it provides a penalty if model liklihoods for non-matching words are too similar to model likelihoods for the correct word in the word set.

Question 2: Compare and contrast the possible advantages and disadvantages of the various model selectors implemented.

Answer 2: A big benefit of using log likelihood with cross-validation is that you get a good idea how well the model works on future data. The data is split up into different folds and the model can actually be tested on previously unseen data which is relatively close to the real-world case of applying it to completely new data. A drawback of this approach is that high complexity (overfitted) models are preferred by the log likelihood and only rejected due to the cross-validation. If all of the data is very similar in appearance, splitting it up into different folds won't help as much to avoid overfitting.

The BIC criterion has the advantage of inherently penalising high complexity models that are prone to overfitting thus allowing for a more general model that should perform well on future data. The drawback here is that the term penalising high complexity models is the only factor helping the model to generalise. It does not explicitly look at unseen data and it does not take into account other words than the one it is currently creating a model for.

A big advantage of the DIC criterion is that it does not look at one word as a separate entity, but rates it with respect to the other words in the training set. A word does not just need to be represented well but it also needs to be represented in a way that significantly differentiates it from other words. This approach is very good for being able to distinguish between words. A drawback is that there is no penalty for overfitting leading to higher complex models being preferred over more general ones.

Question 3: Summarize the error results from three combinations of features and model selectors. What was the "best" combination and why? What additional information might we use to improve our WER? For more insight on improving WER, take a look at the introduction to Part 4.

Answer 3: Looking at what can be achieved with using only one feature set is interesting and forms a baseline for our other tests. Using the ground coordinates combined with the log likelihood + cross validation selector gives the best result of WER=0.53. This is already a very decent result showing that simply using the position of the hand with respect to the nose (thus reducing the effect of camera and/or full body movement) gives us a good idea about the sign. We can improve upon this result by additionally using the delta feature, looking at movement rather than just hand position, and BIC as the selector. This gives us a WER of 0.47.

The same WER of 0.47 is achieved in a very different way by combining the normalised polar coordinates (similar information content as the ground coordinates) for the absolute hand position with the difference in hand position to put a stronger emphasis on how far apart or close together the hands are (and using the DIC selector).

The best result of WER=0.44 is gained by additionally looking at the delta feature of the polar coordinates and using the BIC selector. In this feature combination we get information about the position of the hands (normalised polar coordinates), the movement of the hands (delta), and if we are looking at wide or narrow type of sign (difference in hand position), making it easier to group the sign we are looking at. It seems like adding more feature can help, as long as the information added is either significantly different or helps to put the emphasis on a particularl area of the signing.

However, these models are still trying to read the sign without any context. This is similar to looking at a picture of a landmark without any background information or knowledge about its location - it is doable (especially in case of for example the white house or the Eiffel tower) but not always particularly easy (try the town hall of any mid sized European city). Adding background information (such as the likelihood of a word appearing in a specific language) or information about the context (words before and after) should increase the ability to identify the correct word significantly.

PART 4: (OPTIONAL) Improve the WER with Language Models¶

We've squeezed just about as much as we can out of the model and still only get about 50% of the words right! Surely we can do better than that. Probability to the rescue again in the form of statistical language models (SLM). The basic idea is that each word has some probability of occurrence within the set, and some probability that it is adjacent to specific other words. We can use that additional information to make better choices.

import itertools

# create a DataFrame of log likelihoods for the test word items
df_probs = pd.DataFrame(data=probabilities)
df_probs.head()

probs = df_probs.copy()

n_highest = 4
nbest_words = {n: probs.T[col].nlargest(n_highest).index.tolist() for n, col in enumerate(probs.T)}
nbest_probs = {n: probs.T[col].nlargest(n_highest).tolist() for n, col in enumerate(probs.T)}

alpha = 1
beta_n1 = 0
beta_n2 = 0
beta_n3 = 10

counter = 0
sentences = []
results = []

# loop through the sentences
for nwords in test_set.sentences_index.values():
    sentence_len = len(nwords)
    
    sentence = []
    sentence_probs = []
    
    # loop through the number of words in this sentence
    for iword in range(sentence_len):
        # get the n best guesses for every word in the sentence
        sentence.append(nbest_words[counter])
        
        # get the HMM likelihood for each word added to the sentence
        sentence_probs.append(nbest_probs[counter])
        counter += 1
        
    # form all possible combinations for sentences made up from all n best guesses
    sentences = [' '.join(s) for s in itertools.product(*sentence)]
    sentences_prob = [sum(s) for s in itertools.product(*sentence_probs)]
    sentences_prob = [x*alpha for x in sentences_prob]
    
    # loop through all possible sentences
    for n in range(len(sentences)):
        words = sentences[n].split(" ")
        sentences_prob[n] /= len(words)
        
        # add the likelihood from statistic language model
        # 1-gram
        if beta_n1 != 0:
            for m in words:
                try:
                    sentences_prob[n] += slm_n1.log_p(m) * beta_n1 / len(words)
                except:
                    try:
                        sentences_prob[n] += slm_n1.log_p(m[:-1]) * beta_n1 / len(words)
                    except:
                        pass
                
        # 2-gram
        if beta_n2 != 0:
            for m in range(len(words)-1):
                phrase = " ".join(words[m:m+2])

                try:
                    sentences_prob[n] += slm_n2.log_p(phrase) * beta_n2 / (len(words)-1)
                except:
                    try:
                        # remove numbers
                        phrase = ''.join([i for i in phrase if not i.isdigit()])
                        sentences_prob[n] += slm_n2.log_p(phrase) * beta_n2 / (len(words)-1)
                    except:
                        pass
        # 3-gram
        if beta_n3 != 0:
            for m in range(len(words)-2):
                phrase = " ".join(words[m:m+3])

                try:
                    sentences_prob[n] += slm_n3.log_p(phrase) * beta_n3 / (len(words)-2)
                except:
                    try:
                        # remove numbers
                        phrase = ''.join([i for i in phrase if not i.isdigit()])
                        sentences_prob[n] += slm_n3.log_p(phrase) * beta_n3 / (len(words)-2)
                    except:
                        pass
        
        
    results += sentences[sentences_prob.index(max(sentences_prob))].split(" ")

show_errors(results, test_set)
